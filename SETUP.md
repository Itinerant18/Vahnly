# Drivers-for-u — Local Development Setup

> **Audience:** A new developer who has just cloned the repo and wants a
> working stack on their laptop in under 15 minutes.

This guide is the single source of truth for local setup. The README is a
project orientation; **this document is what you actually run.**

There are two supported paths:

| Path | Best for | Infra |
| --- | --- | --- |
| **A. Docker Compose** (recommended) | First run, macOS/Windows/Linux, no k8s needed | Postgres, Redis 6-shard, Kafka KRaft, Triton, all 11 Go services |
| **B. Local Kubernetes** (`run_e2e_test.ps1`) | Reproducing the prod-shape deploy | Kind/Docker Desktop k8s |

If you don't already have a preference, **use Path A.**

## 2. Configure environment (one-time)

Copy the template and edit secrets:

```bash
cp .env.example .env       # macOS/Linux
Copy-Item .env.example .env   # Windows PowerShell
```

Open `.env` and replace the dev-only placeholders. **Every value here is
consumed by `docker-compose.yml`.** Defaults are wired so a vanilla clone
will run end-to-end — you only need to change them for production-like
testing.

```dotenv
# --- Postgres ---
POSTGRES_USER=postgres
POSTGRES_PASSWORD=password
POSTGRES_DB=delivery_platform
DATABASE_URL=postgres://postgres:password@localhost:5432/delivery_platform?sslmode=disable

# --- Kafka (host-port 19092 is the EXTERNAL listener) ---
KAFKA_BROKERS=localhost:19092

# --- Redis (6-shard cluster) ---
REDIS_CLUSTER_NODES=127.0.0.1:6379,127.0.0.1:6380,127.0.0.1:6381,127.0.0.1:6382,127.0.0.1:6383,127.0.0.1:6384
# The IP map is auto-generated by bootstrap; you only need it for the
# bare-binary k8s path (Path B).

# --- Triton (XGBoost ETA corrector) ---
TRITON_SERVER_URL=127.0.0.1:8001

# --- Auth ---
JWT_SECRET=change-me-in-prod-this-is-a-dev-only-hs256-key
# 32+ bytes; used by the gateway AND the admin handler.

# --- Regions (comma-separated) ---
SUPPORTED_REGIONS=KOL,BLR
DEFAULT_CITY_PREFIX=KOL
```

## 3. Path A — Docker Compose (recommended)

### 3.1 One-shot bootstrap

From the repo root:

**Windows (PowerShell 7):**
```powershell
pwsh ./scripts/bootstrap.ps1
```

**macOS/Linux (bash):**
```bash
chmod +x ./scripts/bootstrap.sh
./scripts/bootstrap.sh
```

What the script does, in order:

1. Verifies Go, Docker, Docker Compose, Node, PowerShell versions.
2. Loads `.env` into the current shell.
3. Runs `docker compose down -v` (idempotent — destroys any prior state).
4. Runs `docker compose up -d --build` (infra first, then services via
   `depends_on` health-checks: `db-migrator` → `ingestion` → `matching-engine`
   → `surge-engine` → `pricing-sync` → `telemetry-pruner` → `offer-expiry-janitor`
   → `outbox-notification-engine` → `public-gateway` → `spatial-analytics`
   → `reconciler-daemon` → `fleet-rebalancer`).
5. Waits up to 90s for the `db-migrator` to report `service_completed_successfully`.
6. Prints the connection map (Postgres `:5432`, Kafka `:19092`, Redis `:6379..6384`,
   Triton `:8001`, Gateway `:8080`, Analytics SSE `:8089`).

### 3.2 Verify the stack

```bash
# All 5 infra health checks must return healthy
docker compose ps

# Hit the gateway health endpoint
curl http://localhost:8080/health
# {"status":"healthy","uptime":"42.123s"}

# Hit the gateway readiness probe (deep-checks Postgres + Redis + Kafka)
curl http://localhost:8080/ready
# {"ready":true,"checks":{"postgres":"OK","redis":"OK","kafka":"OK"}}
```

### 3.3 Seed an end-to-end smoke

```bash
# Apply the canonical seed (KOL region, 1 driver, 1 order in CREATED state)
psql "postgres://postgres:password@localhost:5432/delivery_platform?sslmode=disable" \
     -f bin/seed.sql
```

Then drive traffic with the simulator:

```bash
go run ./cmd/simulator
```

In another terminal, open a WebSocket to the dispatch stream:
```bash
# PowerShell: requires no extra dep; PowerShell has WebClient built-in.
# Easiest: open the admin dashboard (next section) and watch the radar.
```

### 3.4 Run the Go integration test

```bash
export DATABASE_URL="postgres://postgres:password@localhost:5432/delivery_platform?sslmode=disable"
export REDIS_CLUSTER_NODES="127.0.0.1:6379"
export KAFKA_BROKERS="localhost:19092"
export REDIS_IP_MAP=""   # Docker path resolves containers directly

go test -v -tags=integration ./test/integration/...
```

Expected: `8 STAGE * OK` log lines, ending with the SUCCESS banner.

## 4. Path B — Local Kubernetes (advanced)

Use this when you want to mirror the prod-shape deploy (StatefulSets,
headless service, cluster IP routing, KRaft single-broker).

```powershell
# Docker Desktop: enable "Kubernetes" in Settings, wait for green
kubectl get nodes   # must show a single Ready node

# Apply the local topology (namespace + Postgres + Kafka + Redis)
kubectl apply -f deploy/local/local-dev-topology.yaml

# Wait for Postgres + Redis pods
kubectl get pods -n dispatch -w

# Establish port-forwards and print env block
pwsh ./bin/start-port-forwards.ps1
#   ↑ Prints REDIS_IP_MAP plus DATABASE_URL/REDIS_CLUSTER_NODES/KAFKA_BROKERS
#     Copy those into the current PowerShell.

# Build the Go binaries once
pwsh ./scripts/build-services.ps1

# Run the E2E harness
pwsh ./run_e2e_test.ps1
```

> **Why is there a second `local-dev-topology.yaml` at the repo root?**
> The root file is the older v1 manifest and is **kept only for reference**.
> The canonical k8s topology is `deploy/local/local-dev-topology.yaml`.
> Treat the root file as deprecated.

## 8. Common pitfalls

### 8.1 `Redis: MOVED 0 172.28.1.x:6379`
Docker Compose returns container-internal IPs on `MOVED` redirects. The
Go client uses a custom `Dialer` and `REDIS_IP_MAP` to rewrite them.
On the Docker path you can leave `REDIS_IP_MAP` empty — the bootstrap
script prints a sane default. On the k8s path you must set it; the
`bin/start-port-forwards.ps1` helper prints the right value.

### 8.2 `dispatch refuses to start: graph CSV not found`
The dispatch service needs pre-built Contraction Hierarchies. The CSV
files live in `data/` and are NOT in the repo (gitignored). Run the
preprocessor first:
```bash
# 1. Download a city extract (≈100 MB) into data/kolkata_full.osm.pbf
curl -L "https://download.geofabrik.de/asia/india/kolkata-latest.osm.pbf" \
     -o data/kolkata_full.osm.pbf

# 2. Build the contracted graph
mkdir -p data
go run ./cmd/osm-preprocessor
# → emits data/kolkata_nodes.csv + data/kolkata_edges.csv
```
**Without this, dispatch will start but every `ComputeShortestPathETA`
returns 0 and matching is effectively random.** If you don't want OSM
data right now, just point at the placeholder paths — the matcher
still runs, ETA is degenerate, but the rest of the pipeline works:
```bash
export OSM_NODES_DATA_PATH=nonexistent_nodes.csv
export OSM_EDGES_DATA_PATH=nonexistent_edges.csv
```

### 8.3 `Migrations already at latest version` but tables are empty
The `db-migrator` service is the source of truth. If you bypassed it,
run it explicitly:
```bash
docker compose run --rm db-migrator
# or
go run ./cmd/migrate
```

### 8.4 WebSocket disconnects every ~30s
`gateway.delivery.http` is configured with `WriteTimeout: 10s` for
non-WS endpoints. The WS upgrade path holds the connection open; the
issue is usually the Vite dev-server proxy timeout, not the Go server.
Bump `vite.config.ts` `server.proxy[].ws` to `timeout: 0` if needed.

### 8.5 Two pricing services competing
`internal/pricing/service/order_pricing_service.go` exports **two**
surge-sync loops: `StartSurgeMatrixSync` (12-hour TTL) and
`StartSurgeMatrixSyncLoop` (60-second TTL). **Only the 60s loop is
wired** in `cmd/pricing/main.go`. The 12h variant is dead code pending
cleanup. Don't call it.

### 8.6 `frontend` build fails with "Cannot find module 'vite'"
Your `frontend/node_modules/` is stale or partially installed. Run:
```bash
cd frontend
rm -rf node_modules package-lock.json
npm install
```

## 9. Test workflow (everyday developer loop)

```bash

```

## 11. Security notes (must-read)

These are dev-only. **Never deploy as-is.**

1. **`database/migrations/000033_add_system_admins.up.sql`** contains a
   bcrypt hash for the password `Aniket018` (SUPER_ADMIN, region KOL).
   This is for local smoke tests only. **Rotate before any non-dev env**
   by deleting the seed insert and running an interactive admin bootstrap.

2. **`docker-compose.yml` line 430** (`JWT_SECRET_SIGNING_KEY=...`)
   hard-codes a JWT signing key. Override via `.env` and ensure the
   override propagates (`environment:` block in compose reads the host
   env, so set `JWT_SECRET_SIGNING_KEY` in `.env`).

3. **`test/integration/dispatch_e2e_test.go` line 436** hard-codes the
   payment-webhook HMAC secret. Test-only.

4. **`internal/gateway/middleware/region_router.go`** accepts a
   `city_prefix` query param as fallback when `X-Region-Prefix` is
   absent. This is a known leak; remove before prod.

5. **CORS** in `internal/gateway/middleware/cors.go` is wildcard. Lock
   to known admin origins in prod.

## 12. Troubleshooting matrix

| Symptom | Cause | Fix |
| --- | --- | --- |
| `dial tcp 127.0.0.1:5432: connect: connection refused` | Postgres not up | `docker compose up -d spatial-db && docker compose ps` |
| `redis cluster: all 16379/6379 unreachable` | Redis cluster not bootstrapped | Wait for `redis-cluster-orchestrator` to report `service_completed_successfully` (1–2 min) |
| `tritonserver: model not found: xgboost_spatial_corrector` | Volume mount empty | Confirm `model_repository/` is non-empty in repo root; restart `triton-server` |
| `dispatch: panic: nil pointer dereference in disruptWithTripEvent` | `cmd/simulator/main.go` bug | Use the new simulator branch or skip the chaos toggle |
| `gateway: 401 invalid_token` | JWT_SECRET changed between gateway restart and admin login | Re-login (tokens are HS256-signed with current key) |
| `orders row stuck in ASSIGNED` | Reconciler not running | `docker compose up -d reconciler-daemon` |
| `Analytics SSE shows no events` | `CITY_PREFIX` mismatch | Both `cmd/analytics/main.go` and your driver stream must use the same prefix |
| `telemetry stream errors: PERMISSION_DENIED` | Region prefix not in `SUPPORTED_REGIONS` | Add to `.env`: `SUPPORTED_REGIONS=KOL,BLR,HWH` and restart gateway |
